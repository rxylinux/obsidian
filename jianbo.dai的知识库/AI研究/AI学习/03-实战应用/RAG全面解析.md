## 一、什么是 RAG

### 1.1 核心定义

**RAG (Retrieval-Augmented Generation)** = 检索增强生成

```
传统 LLM:
用户问题 → [LLM] → 回答
问题：依赖训练时学到的知识（可能过时、有幻觉）

RAG:
用户问题 → [检索器] → 相关文档 → [LLM + 文档] → 带引用的回答
优势：利用外部知识库，实时、准确、可追溯
```

### 1.2 本质理解

```
RAG = 开卷考试

LLM (闭卷): 只能靠记忆回答
RAG (开卷): 可以查阅资料后再回答

优势：
✅ 知识实时更新（修改文档库即可）
✅ 回答可追溯（标注来源）
✅ 减少幻觉（基于事实回答）
✅ 领域知识注入（无需训练）
```

## 二、RAG vs Fine-tuning

### 2.1 对比表格

| 维度 | RAG | Fine-tuning | RAG + Fine-tuning |
|:-----|:-----|:------------|:------------------|
| **知识更新** | ✅ 实时更新文档 | ❌ 需要重新训练 | ✅ 两者结合 |
| **领域适应** | ✅ 快速注入知识 | ✅ 学习领域风格 | ✅ 最佳效果 |
| **成本** | ✅ 低 | ❌ 高（GPU、数据） | 中 |
| **幻觉** | ✅ 减少 | ⚠️ 可能增加 | ⚠️ 需要控制 |
| **可解释性** | ✅ 可追溯来源 | ❌ 黑盒 | ✅ 部分可解释 |
| **适用场景** | 知识密集型任务 | 风格/格式适应 | 复杂应用 |

### 2.2 选择决策

```
你的需求是？

├─ 知识实时更新
│   └─ RAG ✅
│
├─ 领域知识注入（医疗、法律等）
│   └─ RAG ✅
│
├─ 特定风格/格式输出
│   └─ Fine-tuning ✅
│
├─ 私有数据保护
│   └─ RAG ✅
│
├─ 模型行为定制
│   └─ Fine-tuning ✅
│
└─ 追求极致效果
    └─ RAG + Fine-tuning ✅
```

### 2.3 实际应用建议

```
80% 的场景：只用 RAG
  - 企业知识库问答
  - 技术文档助手
  - 客服系统
  - 研究资料分析

15% 的场景：Fine-tuning
  - 特定格式输出（JSON、SQL）
  - 品牌风格对齐
  - 代码生成

5% 的场景：RAG + Fine-tuning
  - 医疗诊断助手（领域知识 + 医学风格）
  - 法律顾问（法律条文 + 法律推理风格）
```

## 三、RAG 解决的问题

### 3.1 LLM 的局限性

```
LLM 四大问题：

1. 知识截止
   训练后学到某天为止
   → 无法回答新事件

2. 幻觉
   自信地胡说八道
   → 误导性回答

3. 领域知识不足
   通用训练数据有限
   → 专业领域表现差

4. 黑盒不可追溯
   不知道答案来源
   → 难以验证
```

### 3.2 RAG 的解决方案

| 问题 | RAG 解决方案 |
|:-----|:------------|
| **知识截止** | ✅ 更新文档库即可，无需训练 |
| **幻觉** | ✅ 基于检索到的文档回答 |
| **领域知识** | ✅ 注入领域文档 |
| **不可追溯** | ✅ 标注答案来源 |

## 六、工作流程详解

> [!info] RAG 三阶段总览
> RAG 系统分为三个阶段：**准备阶段（离线）**、**查询阶段（在线）**、**训练阶段（RAG 2.0）**
>
> ```
> 【准备阶段】（离线）
> 文档 → [Embedding/分词] → 向量/倒排索引 → 存储到数据库
>
> 【查询阶段】（在线）
> 用户查询 → [查询优化器] → [检索器] → 检索数据 → [生成器] → 答案
>
> 【训练阶段】（RAG 2.0 特有）
> 联合训练检索器 + 生成器 → 梯度双向流动 → 协同进化
> ```

#### 对应关系表

| 准备阶段操作 | 生成数据类型 | 查询阶段检索器 | 说明 |
|:------------|:------------|:-------------|:-----|
| **Embedding 向量化** | 向量 | **Dense 检索器** | 使用向量相似度检索 |
| **分词 Tokenization** | 倒排索引 | **BM25 检索器** | 使用词项匹配检索 |
| **两者都做** | 向量 + 倒排 | **混合检索器** | 结合两者优势 |

#### 代码示例

```python
# ========== 准备阶段：决定生成什么数据 ==========

# 选项1: 只做向量化 → 只能用 Dense 检索器
doc_vectors = embedder.encode(documents)
vector_store.add(documents, doc_vectors)

# 选项2: 只做分词 → 只能用 BM25 检索器
tokenized_docs = [doc.split() for doc in documents]
bm25 = BM25Okapi(tokenized_docs)

# 选项3: 两者都做 → 可以用混合检索器
doc_vectors = embedder.encode(documents)
tokenized_docs = [doc.split() for doc in documents]
bm25 = BM25Okapi(tokenized_docs)
vector_store.add(documents, doc_vectors)


# ========== 查询阶段：检索器必须对应 ==========

# 对应选项1: Dense 检索器（需要向量数据）
dense_retriever = DenseRetriever(vector_store)

# 对应选项2: BM25 检索器（需要倒排索引）
bm25_retriever = BM25Retriever(bm25)

# 对应选项3: 混合检索器（需要两者都有）
hybrid_retriever = HybridRetriever(
    dense_retriever=dense_retriever,
    bm25_retriever=bm25_retriever
)
```

#### 设计原则

```
正确的 RAG 设计顺序：

1. 先决定需求（性能、成本、场景）
2. 再决定用什么检索器
3. 最后决定准备阶段生成什么数据

自底而上的依赖：
   数据存储 → 检索方式 → 检索器

类比：
   准备阶段 = 建图书馆（建书架、分类系统）
   查询阶段 = 图书管理员找书

   有什么工具才能干什么活！
```

### 6.2 在线阶段（查询响应）

```python
# 伪代码
def rag_pipeline(query):
    # 1. 查询优化
    optimized_query = query_optimizer.rewrite(query)

    # 2. 检索
    candidates = retriever.retrieve(optimized_query, top_k=100)

    # 3. 重排序
    reranked_docs = reranker.rerank(query, candidates, top_n=10)

    # 4. 上下文构建
    context = context_builder.build(reranked_docs)

    # 5. 生成
    prompt = f"""根据以下上下文回答问题：

上下文：
{context}

问题：{query}

回答："""

    answer = llm.generate(prompt)

    # 6. 后处理
    return {
        "answer": answer,
        "sources": reranked_docs
    }
```

## 十一、Reranker 模型

### 11.1 为什么需要 Reranker

```
两阶段检索：

第一阶段：召回 (Recall)
  目标：快速找出大量可能相关的文档
  方法：向量检索、BM25
  返回：Top-100 候选

第二阶段：精排 (Rerank)
  目标：精确排序，找出最相关的
  方法：交叉编码器 (Cross-Encoder)
  返回：Top-5 最终文档

为什么？
  - 向量检索快但不够精确
  - 交叉编码器精确但慢
  - 结合起来：又快又准！
```

### 11.2 主流 Reranker

| 模型 | 类型 | 特点 |
|:-----|:-----|:-----|
| **BGE-Reranker-v2** | 开源 | 中文友好、效果好 |
| **Cohere Rerank 3** | 商业 API | 质量最高 |
| **Jina Reranker v2** | 开源 | 支持长文本 |
| **ColBERT** | 开源 | Token 级精细匹配 |

## 十三、查询优化

### 13.1 查询改写 (Query Rewriting)

```python
# 用 LLM 改写查询，提高检索质量

原始查询: "苹果"
  → 模糊：水果？公司？

改写后:
  - "苹果公司的最新产品"
  - "iPhone 16 功能介绍"
  - "苹果营养价值"

→ 检索结果更准确！
```

### 13.2 HyDE (Hypothetical Document Embeddings)

```
核心思想：先生成假设答案，再用假设答案去检索

Query: "RAG 的优势是什么？"

步骤1: LLM 生成假设答案
  "RAG 的优势包括：知识实时更新、减少幻觉、可追溯来源..."

步骤2: 用假设答案检索
  （假设答案包含更多相关关键词）

步骤3: 检索到更相关的文档！

效果：召回率提升 10-20%
```

### 13.3 查询分解 (Query Decomposition)

```
复杂问题 → 多个简单子问题

Query: "Transformer 和 RNN 的区别是什么？它们各自适用于什么场景？"

分解:
  - "Transformer 的架构特点"
  - "RNN 的架构特点"
  - "Transformer vs RNN 对比"
  - "Transformer 适用场景"
  - "RNN 适用场景"

→ 分别检索，结果更全面
```

## 十四、检索优化

### 14.1 混合检索 (Hybrid Search)

```python
# 结合语义和关键词检索

final_score = α × dense_score + (1-α) × bm25_score

# α = 0.5: 各占一半
# α = 0.7: 更重语义
# α = 0.3: 更重关键词

效果：
  - 精确术语匹配 (BM25)
  - 语义理解 (Dense)
  - 互补优势，效果最佳
```

### 14.2 递归检索 (Recursive Retrieval)

```
文档层级检索：

文档
  ├─ 章节
  │   ├─ 小节
  │   │   └─ 段落 ← 精细检索
  │   └─ 段落
  └─ 段落

流程：
  1. 先检索最相关的段落
  2. 向上扩展到完整上下文
  3. 返回有层级关系的文档
```

### 14.3 自动合并检索

```
问题：切分后的片段可能失去上下文

解决：检索后自动合并相邻片段

检索到: [片段3], [片段5], [片段7]

合并后:
  - [片段2-4] ← 合并上下文
  - [片段5-6]
  - [片段7-8]

→ 更完整的上下文
```

## 十五、生成优化

### 15.1 Prompt 设计

```python
# 好的 RAG Prompt

template = """你是一个专业的知识助手。

请根据以下上下文信息回答用户的问题。如果上下文中没有相关信息，请明确说明"根据现有信息无法回答"。

上下文信息：
{context}

问题：{question}

要求：
1. 答案必须基于给定的上下文
2. 标注答案来源（文档名称、页码等）
3. 如果上下文矛盾，请说明
4. 保持简洁明了

回答："""
```

### 15.2 上下文压缩

```
问题：检索到的文档太长，超出 LLM 上下文窗口

解决：只保留最相关的部分

方法：
1. 按相似度截取 Top-N
2. 提取关键句（摘要）
3. LLM 重新生成精简版上下文

效果：
  - 减少 Token 使用
  - 提高答案质量
  - 降低成本
```

## 十六、RAG 2.0

### 16.1 核心概念

```
RAG 1.0: 检索器和生成器独立训练
  → 各自优化，组合次优

  → 生成器损失回传给检索器

## 十七、GraphRAG

### 17.1 核心思想

```
传统 RAG:
  文档 → 切分 → 向量检索
  问题：丢失文档间关系

GraphRAG:
  文档 → 提取实体和关系 → 知识图谱 → 图检索
  优势：保留全局关系，适合跨文档分析
```

### 17.2 适用场景

| 场景 | 传统 RAG | GraphRAG |
|:-----|:---------|:---------|
| 单文档问答 | ✅ | ✅ |
| 跨文档关系分析 | ❌ | ✅✅✅ |
| 全局性总结 | ❌ | ✅✅✅ |
| 实体关系查询 | ⚠️ | ✅✅✅ |
| Token 成本 | 1x | 5-10x |

## 十八、Agentic RAG

### 18.1 核心概念

```
传统 RAG: 固定流程
  Query → Retrieve → Generate

Agentic RAG: Agent 动态决策
  Query → [Agent 判断] →
    ├─ 需要检索？→ Retrieve → Generate
    ├─ 需要多步？→ 多轮检索 → Generate
    └─ 不需要检索？→ 直接 Generate

优势：
  - 自主决策
  - 多步推理
  - 工具调用
```

### 18.2 与其他概念区别

| 特性 | Agentic RAG | Adaptive RAG | RAG 2.0 |
|:-----|:------------|:-------------|:--------|
| Agent 控制 | ✅ | ✅ | ❌ |
| 动态决策 | ✅ | ✅ | ❌ |
| 联合训练 | ❌ | ❌ | ✅ |

## 二十四、快速参考

### 24.1 核心要点

```
RAG = 开卷考试的 LLM

核心组件：
  检索器 + Reranker + 生成器

最佳实践：
  Hybrid 检索 + Reranker + 好的 Prompt

关键公式：
  高质量 RAG = 优质数据 + 合理架构 + 持续优化

常见错误：
  ❌ 切分太大/太小
  ❌ 只用向量检索
  ❌ 没有重排序
  ❌ Prompt 设计差
```

### 24.2 一句话总结

> **RAG 让 LLM 能够查阅资料后再回答，就像开卷考试一样，知识实时、准确、可追溯。**
